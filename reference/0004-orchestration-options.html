Snowflake and dbt Orchestration Options

A quick decision reference for Snowflake on Azure with part of the SQL transformation layer in dbt.

Recommended Default

Keep CI/CD in GitHub Actions or Azure DevOps. Use dbt Cloud jobs for dbt model builds. Use Snowflake Tasks for simple in-Snowflake schedules and task graphs. Use Azure Data Factory when Azure-native ingestion or enterprise scheduling is already the control plane.

Option Matrix

Snowflake Tasks
Best for SQL, stored procedures, streams, task graphs, and workloads that should live entirely inside Snowflake.
dbt Cloud Jobs
Best for scheduled or event-triggered dbt builds, dbt artifacts, job history, model logs, and dbt-owned dependencies.
Azure Data Factory
Best for Azure-native ingestion, copy activity, enterprise triggers, and pipelines that span Azure services plus Snowflake.
GitHub Actions
Best for CI/CD: tests, packaging, deploys, and manual release workflows. Not ideal as the main recurring data scheduler.
Azure DevOps Pipelines
Best if the organization standardizes on Azure DevOps for repos, approvals, environments, and releases.
Airflow
Best for mature DAG orchestration across many systems when the team can operate Airflow reliably.
Dagster
Best for asset-aware orchestration, especially when dbt models, Python assets, and lineage should be modeled together.
Prefect
Best for Python-first flows with flexible runtime behavior and simpler operational ergonomics than heavy DAG platforms.

Selection Rules

  • If the dependency is between dbt models, let dbt own it.
  • If the dependency is inside Snowflake and close to data, consider Snowflake Tasks.
  • If the dependency starts in Azure storage, Event Grid, Data Factory, Synapse, or enterprise Azure scheduling, consider Azure Data Factory.
  • If the dependency is deploy-time, not run-time, use GitHub Actions or Azure DevOps.
  • If the workflow spans many systems and needs DAG-level observability, evaluate Airflow, Dagster, or Prefect.
