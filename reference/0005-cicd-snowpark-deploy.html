CI/CD Snowpark Deploy Reference

A compact map of the GitHub Actions setup for this Snowpark project.

CI Workflow

.github/workflows/ci.yml validates code. It does not deploy files to Snowflake.
  1. Check out repository.
  2. Set up Python 3.12.
  3. Install requirements-ci.txt.
  4. Run python -m pytest tests.
  5. Load Snowflake secrets into environment variables.
  6. Create temporary ~/.snowflake/connections.toml.
  7. Run python src/smoke_connection.py.

CD Workflow

.github/workflows/deploy-workspace.yml validates code and copies selected source files to Snowflake workspace folders.
  1. Manual trigger with workflow_dispatch.
  2. Install Python dependencies and Snowflake CLI.
  3. Run unit tests.
  4. Require all Snowflake secrets.
  5. Create temporary Snowflake connection config.
  6. Run smoke test.
  7. Upload files with snow stage copy ... --overwrite.

Secrets Boundary

  • Secrets live in GitHub repository settings.
  • The workflow reads them with ${{ secrets.SECRET_NAME }}.
  • The runner receives them as environment variables for that job.
  • The temporary connection file exists only on the GitHub-hosted runner.
  • Never commit connections.toml or plaintext passwords.

Command Cheatsheet

# Run CI manually
gh workflow run CI --repo emilyg888/snowpark-api-python --ref main

# List latest CI runs
gh run list --workflow="ci.yml" --limit 5

# View a run
gh run view RUN_ID

# Run workspace deploy manually
gh workflow run deploy-workspace.yml --repo emilyg888/snowpark-api-python --ref main

# Watch latest deploy
gh run watch RUN_ID

Decision Rules

  • If it proves code behavior, put it in CI.
  • If it publishes code or artifacts, put it in CD.
  • If it schedules production data work, use orchestration instead of CI/CD.
  • If it needs Snowflake credentials in GitHub Actions, use repository secrets.
