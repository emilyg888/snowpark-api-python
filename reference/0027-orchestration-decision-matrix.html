Orchestration Decision Matrix

A compact reference for choosing how to run the Snowpark, SQL, dbt, and Snowpark ML work from Lessons 1-26.

Recommended Stack

GitHub Actions = code checks, CI, repo hygiene
dbt Cloud = production transformation/test jobs for dbt marts
Snowflake tasks = Snowflake-native SQL/procedure schedules
Airflow/Dagster/Prefect = enterprise cross-system orchestration

Decision Table

Option Best For Tradeoff
dbt Cloud jobs Building and testing dbt models in Snowflake. Excellent for SQL transformation DAGs; weaker for non-dbt Python/API workflows.
Snowflake tasks Scheduling SQL, stored procedures, and in-Snowflake pipelines. Close to data; less convenient for Git-based multi-tool orchestration.
GitHub Actions CI, linting, unit tests, smoke tests, deployment scripts. Good for code lifecycle; not ideal as the main data scheduler.
Airflow Enterprise batch DAGs across many systems. Flexible and mature; needs platform operation and DAG discipline.
Dagster Asset-oriented orchestration with dbt and ML lineage. Strong data asset model; new platform concepts to learn.
Prefect Python-first workflows and lightweight orchestration. Ergonomic for Python; governance depends on deployment setup.

Course Mapping

Lessons 1-12: GitHub Actions for local/unit tests; Snowflake tasks only after scripts are productionized.
Lessons 13-15: Snowflake SQL/procedure/UDF/external-function objects; deploy by GitHub Actions or Snowflake CLI.
Lesson 16: dbt or Snowflake SQL; dbt is preferred once the star schema becomes a maintained mart.
Lessons 17-21: Snowpark ML workflows; use external orchestrator if training, registry, and monitoring must coordinate.
Lessons 22-26: dbt Cloud deployment job is the primary orchestrator.

Rule Of Thumb

If the work is mostly dbt models and tests, use dbt Cloud. If the work is a Snowflake-native SQL or stored-procedure schedule, use Snowflake tasks. If the work crosses files, APIs, ML, dbt, and cloud services, use Airflow, Dagster, or Prefect.
