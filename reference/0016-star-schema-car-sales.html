Car Sales Star Schema

Quick reference for modeling CAR_SALES as dimensions and facts.

Design Rules

  • Declare the grain before creating tables.
  • Put descriptive attributes in dimensions.
  • Put transaction measures and dimension keys in facts.
  • Use a separate bridge or factless table when an array creates multiple child rows.

Tables

DIM_SALES_PERSONS
SALES_PERSON_ID, SALES_PERSON_NAME
DIM_CUSTOMERS
CUSTOMER_KEY, name, phone, address
DIM_VEHICLES
VEHICLE_KEY, make, model, year
FACT_VEHICLE_SALES
SALE_KEY, date, dealership, dimension keys, price, count
BRIDGE_VEHICLE_SALE_EXTRAS
SALE_KEY, one extra per row

Query Pattern

select
  v.make,
  v.model,
  s.sales_person_name,
  count(*) as sale_count,
  sum(f.sale_price) as revenue
from fact_vehicle_sales f
join dim_vehicles v on f.vehicle_key = v.vehicle_key
join dim_sales_persons s on f.sales_person_key = s.sales_person_key
group by v.make, v.model, s.sales_person_name
order by revenue desc;

Fact Grain

FACT_VEHICLE_SALES = one source sale + customer + vehicle combination
BRIDGE_VEHICLE_SALE_EXTRAS = one fact sale + extra combination
