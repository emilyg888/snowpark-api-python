Choosing Orchestration for Snowpark, dbt, and ML

Goal: choose a realistic orchestration solution for everything built from Lessons 1-26.

Orchestration is not one tool decision. It is a boundary decision: what runs code checks, what builds data models, what schedules Snowflake-native work, and what coordinates systems outside Snowflake.

The Workload

Lessons 1-12
Snowpark Python scripts, local tests, smoke checks, DataFrame patterns.
Lessons 13-16
Stored procedure, UDF, external function, SQL star-schema model.
Lessons 17-21
Snowpark ML lab work: features, training, registry, monitoring.
Lessons 22-26
dbt Cloud source, staging, marts, tests, deployment job.

The Practical Architecture

GitHub Actions
  -> checks repo quality, unit tests, smoke tests, deploy scripts

dbt Cloud
  -> builds STAGING and MART objects
  -> runs dbt tests
  -> stores job run history and artifacts

Snowflake tasks
  -> schedules Snowflake-native SQL and stored procedures
  -> runs close to the data

Airflow / Dagster / Prefect
  -> coordinates data lake files, APIs, Snowflake, dbt, ML, notifications

Option Comparison

Option Pros Cons Use Here
dbt Cloud jobs Best fit for dbt DAGs, tests, docs, run history. Not a general Python/API orchestrator. Primary for Lessons 22-26.
Snowflake tasks Native scheduling, task graphs, serverless option, close to data. Less natural for repo-driven Python packaging and cloud APIs. Good for stored procedures and SQL refreshes.
GitHub Actions Strong CI/CD, secrets, tests, deployment automation. Weak as a data pipeline scheduler and harder to monitor as a data platform. Use for code checks and deployment.
Airflow Mature enterprise DAG scheduler with many providers. Requires infrastructure and careful DAG maintenance. Use when many external systems must be coordinated.
Dagster Asset lineage, dbt integration, good for data products and ML dependencies. Requires learning asset-oriented design. Use when marts, ML models, and data quality are one asset graph.
Prefect Python-first and ergonomic for scripts, APIs, and Snowflake calls. Less warehouse-native than Snowflake tasks and less dbt-native than dbt Cloud. Use for lightweight Python orchestration.

Recommended Solution

For this course project, use a hybrid solution:

  1. GitHub Actions runs unit tests and smoke checks for Snowpark scripts.
  2. dbt Cloud runs dbt build --select source:car_sales_source+ for the car-sales mart.
  3. Snowflake tasks schedule mature Snowflake-native objects such as stored procedures.
  4. Dagster or Airflow becomes the enterprise orchestrator only when the pipeline spans lake files, API calls, ML training, dbt jobs, and alerts.
