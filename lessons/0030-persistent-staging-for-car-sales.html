Persistent Staging for Car Sales

Goal: turn the current dbt staging views into persisted staging tables that carry raw load metadata forward.

A production staging layer should be re-runnable, inspectable, and traceable. For this project, staging needs to preserve SOURCE_FILE, SOURCE_SALE_KEY, LOAD_BATCH_ID, and LOAD_TS.

Current State

The project currently materializes staging as views in dbt_project.yml:

models:
  car_sales_dbt:
    staging:
      +materialized: view
      +schema: STAGING

That was correct for early dbt learning. For E2E ingestion, make staging persistent so ADF and dbt Cloud runs have stable objects to inspect after a load.

Target Change

Change staging to table materialization first:

models:
  car_sales_dbt:
    staging:
      +materialized: table
      +schema: STAGING

Later, when the design is stable, promote selected staging models to incremental.

Metadata To Carry

Column Reason
SOURCE_FILEShows which ADLS file produced the row.
SOURCE_SALE_KEYIdentifies exact duplicate JSON records.
LOAD_BATCH_IDConnects the row to one ADF pipeline run.
LOAD_TSAllows incremental filters and audit checks.

Base Model Pattern

select
  src,
  source_file,
  source_sale_key,
  load_batch_id,
  load_ts,
  src:date::date as sale_date,
  src:dealership::string as dealership,
  src:salesperson.id::string as sales_person_id,
  src:salesperson.name::string as sales_person_name
from {{ source('car_sales_source', 'CAR_SALES') }}

Array Staging Rule

Every flattened staging model should keep the parent metadata:

select
  base.source_file,
  base.source_sale_key,
  base.load_batch_id,
  base.load_ts,
  vehicle.index::number as vehicle_index,
  vehicle.value:make::string as make
from {{ ref('stg_car_sales_base') }} as base,
  lateral flatten(input => base.src:vehicle) as vehicle
