CI/CD for Snowpark Workspace Deploy

Goal: explain the GitHub Actions path from local Snowpark code to tested code in a Snowflake workspace.

CI answers “does the code still work?” CD answers “can we publish this code to the target environment?”

The Repo Control Plane

Your VS Code codebase now has two GitHub Actions workflows with different jobs.

CI
.github/workflows/ci.yml runs unit tests and a Snowflake smoke test.
CD
.github/workflows/deploy-workspace.yml runs deploy gates, then copies lesson files to the Snowflake workspace.
Secrets
GitHub repository secrets become temporary environment variables inside the runner.
Runner
GitHub creates a clean Ubuntu machine for each run. Its ~/.snowflake/connections.toml is temporary.

CI Steps We Covered

  1. Trigger: run on pushes, pull requests, or manual workflow_dispatch.
  2. Check out repo: download the GitHub repo into the runner workspace.
  3. Set up Python 3.12: match the local Snowpark learning environment closely enough for tests.
  4. Install dependencies: install requirements-ci.txt.
  5. Run unit tests: execute python -m pytest tests.
  6. Inject secrets: map GitHub repository secrets into job environment variables.
  7. Create temporary Snowflake config: generate ~/.snowflake/connections.toml inside the runner.
  8. Run smoke test: execute python src/smoke_connection.py.

CD Steps We Covered

  1. Manual trigger: run deploy-workspace.yml only when you choose to deploy.
  2. Repeat deploy gates: install dependencies, run unit tests, require secrets, configure Snowflake, and smoke test.
  3. Install Snowflake CLI: make the snow command available in the runner.
  4. Copy files: run snow stage copy for LESSON1.py, LESSON2.py, and LESSON3.py.
  5. Verify logs: confirm the GitHub Action shows each file as uploaded.

Key Mental Model

The runner is not your Mac. It starts empty, receives the repo, receives secrets from GitHub, builds its own temporary Snowflake connection file, runs checks, and then disappears.
