Modeling Car Sales as a Star Schema

Goal: turn the nested CAR_SALES source into dimension tables and fact tables for analysis.

A star schema separates descriptive context into dimensions and measurable events into facts. The first design decision is the fact grain: exactly what one fact row means.

The Grain

For CAR_SALES, use this transaction grain:

One row in FACT_VEHICLE_SALES = one source sale + customer + vehicle combination.

The extras array has a different grain because one sale can have many extras. Keep it in a separate bridge-style fact table:

One row in BRIDGE_VEHICLE_SALE_EXTRAS = one fact sale + extra combination.

Star Model

DIM_SALES_PERSONS
Salesperson id and name from SRC:salesperson.
DIM_CUSTOMERS
Customer name, phone, and address from flattened SRC:customer.
DIM_VEHICLES
Vehicle make, model, and year from flattened SRC:vehicle.
FACT_VEHICLE_SALES
Sale date, dealership, dimension keys, sale price, and vehicle count.
BRIDGE_VEHICLE_SALE_EXTRAS
Sale key plus one extra value per row.

Read This Code

create or replace table DIM_SALES_PERSONS as
select distinct
  SRC:salesperson.id::string as SALES_PERSON_ID,
  SRC:salesperson.name::string as SALES_PERSON_NAME
from SNOWPARK_SAMPLE_DATA.PUBLIC.CAR_SALES;

create or replace table DIM_VEHICLES as
select
  md5(
    upper(coalesce(car.value:make::string, '')) || '|' ||
    upper(coalesce(car.value:model::string, '')) || '|' ||
    coalesce(car.value:year::string, '')
  ) as VEHICLE_KEY,
  car.value:make::string as MAKE,
  car.value:model::string as MODEL,
  car.value:year::number as YEAR
from SNOWPARK_SAMPLE_DATA.PUBLIC.CAR_SALES cs,
  lateral flatten(input => cs.SRC:vehicle) car
where car.value:make is not null
qualify row_number() over (
  partition by VEHICLE_KEY
  order by MAKE, MODEL, YEAR
) = 1;

create or replace table FACT_VEHICLE_SALES as
select
  md5(cs.SRC::string || '|' || customer.index::string || '|' || car.index::string) as SALE_KEY,
  cs.SRC:date::date as SALE_DATE,
  cs.SRC:dealership::string as DEALERSHIP,
  md5(
    upper(coalesce(customer.value:name::string, '')) || '|' ||
    coalesce(customer.value:phone::string, '') || '|' ||
    upper(coalesce(customer.value:address::string, ''))
  ) as CUSTOMER_KEY,
  md5(
    upper(coalesce(car.value:make::string, '')) || '|' ||
    upper(coalesce(car.value:model::string, '')) || '|' ||
    coalesce(car.value:year::string, '')
  ) as VEHICLE_KEY,
  car.value:price::number as SALE_PRICE,
  1 as VEHICLE_COUNT
from SNOWPARK_SAMPLE_DATA.PUBLIC.CAR_SALES cs,
  lateral flatten(input => cs.SRC:customer) customer,
  lateral flatten(input => cs.SRC:vehicle) car;

That pattern creates small descriptive dimensions and a fact table that stores keys plus measures. The sale price belongs in the fact table because it is a transaction measure, not a vehicle description.
