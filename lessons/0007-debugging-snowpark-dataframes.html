Debugging Snowpark DataFrames

Goal: inspect a DataFrame safely before running a large query or wiring it into Streamlit.

Debugging Snowpark means separating three questions: what columns exist, what SQL will run, and what small sample comes back.

The Debugging Ladder

Schema
Check column names and types before assuming downstream code can use them.
Plan
Use explain() to inspect how Snowflake plans the query.
SQL
Inspect generated SQL when Snowpark behavior is unclear.
Sample
Use limit(...).show() to run a small, intentional action.

Read This Code

def debug_orders_with_customers(df):
    print("Schema")
    df.print_schema()

    print("Query plan")
    df.explain()

    print("Generated SQL")
    for query in df.queries["queries"]:
        print(query)

    print("Small sample")
    df.limit(10).show()

This function is intentionally separate from the builder. The builder creates a lazy DataFrame. The debug helper decides when to inspect metadata and when to trigger a small action.

Safe Debugging Rules

  • Use print_schema() before debugging column errors.
  • Use explain() before tuning or questioning how a query runs.
  • Use df.queries when you need to compare Snowpark code with generated SQL.
  • Use limit(...).show() instead of show() on a broad DataFrame.
  • Keep debug helpers out of unit-tested builder functions.
